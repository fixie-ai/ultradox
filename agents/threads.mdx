Threads allow you to spawn independent, parallel conversations that run alongside the main call. Each thread forks from the parent's conversation history and executes autonomously вҖ” generating responses, calling tools, and maintaining its own state вҖ” without interrupting the user-facing conversation.

<Note>
  <b>Advanced Feature</b>

  Threads are designed for use cases that require background processing or parallel reasoning during a live call. For most applications, the main conversation is sufficient. Make sure you're comfortable with [Agents](/agents/overview) and [Data Messages](/apps/datamessages) before diving in.
</Note>

## When to Use Threads

Threads are useful when you need the agent to do work in the background without disrupting the flow of the main conversation. Common scenarios include:

**Background Research** вҶ’ While the agent talks with the user, a thread can query APIs, search databases, or perform analysis and have results ready when needed.

**Parallel Tool Execution** вҶ’ Spawn threads to run multiple tool calls concurrently instead of handling them sequentially in the main conversation.

**Delegated Tasks** вҶ’ Hand off subtasks (e.g., filling out a form, validating data) to a thread while the main conversation continues.

**Speculative Processing** вҶ’ Start work on a likely next step before the user confirms, reducing latency when they do.

## How Threads Work

When you spawn a thread, the system:

1. **Forks the conversation history** вҖ” The thread gets a copy of the parent's full message history at the point of spawning.
2. **Copies tool state** вҖ” The thread inherits a deep copy of the parent's current tool state.
3. **Runs independently** вҖ” The thread enters its own generate-and-act loop, processing messages and invoking tools on its own.
4. **Reports results** вҖ” The thread sends response messages back to the client via [data messages](/apps/datamessages#thread-messages).

Threads do not produce audio output. All thread responses are delivered as text via data messages.

**IDLE** вҶ’ The thread is waiting for messages. If the thread has finished all its work and no further messages arrive, it stays idle.

**GENERATING** вҶ’ The thread is running LLM inference to produce a response.

You can query the current state of all active threads using the [`checkThreadState`](/tools/built-in-tools) built-in tool.

## Spawning a Thread

Threads are created by sending a [`spawnThread`](/apps/datamessages#spawnthread) data message. Here's a basic example:

```js
// Spawn a thread to research a topic in the background
session.sendData({
  type: "spawn_thread",
  newThreadId: "research-task-1",
  additionalMessages: [
    {
      type: "user_text_message",
      text: "Look up the latest pricing for the Enterprise plan and summarize it."
    }
  ]
});
```

The server responds with a [`threadSpawned`](/apps/datamessages#threadspawned) message on success or a [`threadRejected`](/apps/datamessages#threadrejected) message if the thread cannot be created.

### Providing Initial Context

The `additionalMessages` field lets you give the thread its initial instructions. These are appended to the forked conversation history before the thread starts generating. You can use:

- **`user_text_message`** вҖ” Adds a user message that the thread will respond to.
- **`forced_agent_message`** вҖ” Pre-populates agent text or tool calls in the thread's history (e.g., to provide known context without making the thread re-derive it).

```js
session.sendData({
  type: "spawn_thread",
  additionalMessages: [
    {
      type: "forced_agent_message",
      content: "I already know the user's account ID is 12345.",
      knownToolResults: [
        {
          name: "lookupAccount",
          arguments: { email: "user@example.com" },
          result: "{\"accountId\": \"12345\", \"plan\": \"pro\"}"
        }
      ]
    },
    {
      type: "user_text_message",
      text: "Now check their billing history for the last 3 months."
    }
  ]
});
```

### Setting Thread Limits

You can constrain a thread's resource usage with the `limits` field:

```js
session.sendData({
  type: "spawn_thread",
  newThreadId: "bounded-task",
  limits: {
    generationLimit: 5,               // Max 5 LLM generation rounds
    threadOutputTokenLimit: 2000,      // Max 2000 output tokens total
    generationOutputTokenLimit: 500    // Max 500 output tokens per generation
  },
  additionalMessages: [
    { type: "user_text_message", text: "Summarize the conversation so far in 2-3 sentences." }
  ]
});
```

Available limits:

| Limit | Description |
| ----- | ----------- |
| `generationLimit` | Maximum number of LLM generation rounds for the thread. |
| `threadOutputTokenLimit` | Maximum total output tokens across the thread's lifetime. |
| `generationOutputTokenLimit` | Maximum output tokens per single generation. |
| `generationInputTokenLimit` | Maximum input tokens per single generation. |
| `threadFuzzyInputTokenLimit` | Approximate limit on total input tokens (uses fast estimation). |

When any limit is exceeded, the thread terminates and the client receives a `threadTerminated` message with the reason.

### Filtering Available Tools

Use `toolFilter` to restrict which tools a thread can access:

```js
session.sendData({
  type: "spawn_thread",
  toolFilter: {
    allowedTools: ["searchDatabase", "formatResponse"]  // Only these tools
  },
  additionalMessages: [
    { type: "user_text_message", text: "Search for recent orders." }
  ]
});
```

You can specify `allowedTools` (allowlist) or `disallowedTools` (blocklist), but not both.

## Sending Messages to a Thread

After spawning, you can send additional messages to a thread by including `threadId` on a `user_text_message` or `forced_agent_message`:

```js
// Send a follow-up message to an existing thread
session.sendData({
  type: "user_text_message",
  text: "Also check for any pending refunds.",
  threadId: "research-task-1"
});
```

## Receiving Thread Responses

Thread responses arrive as data messages with a `threadId` field. The response flow is:

1. **`sideGenerationDelta`** вҖ” Streaming text deltas as the thread generates (similar to transcript deltas).
2. **`sideGenerationCompleted`** вҖ” The full response text and any tool calls once generation finishes.
3. **`sideGenerationFailed`** вҖ” Sent instead of `completed` if generation fails.

See [Thread Messages](/apps/datamessages#thread-messages) for the full message reference.

## Inter-Thread Communication

Threads can communicate with each other and with the main conversation (the "UI" thread) through tool responses. When a tool returns a `sendToThread` response type, the result is routed to the specified thread as a message. This allows threads to:

- Report findings back to the main conversation
- Delegate subtasks to other threads
- Coordinate multi-step workflows

## Error Handling

Threads can fail for several reasons:

| Reason | Description |
| ------ | ----------- |
| Limits exceeded | The thread hit one of its configured resource limits. |
| Inference failure | The LLM generation failed (e.g., model error). |
| Cancellation | The thread was cancelled, typically because the call ended or a stage change occurred. |
| Duplicate ID | A thread with the same ID already exists (spawn is rejected). |

When a thread fails, the client receives a [`threadTerminated`](/apps/datamessages#threadterminated) message with the `reason` field explaining what happened. Failed threads cannot be resumed вҖ” spawn a new thread if you need to retry.

## Nested Threads

Threads can spawn child threads. A `spawnThread` message sent with a `parentThreadId` pointing to an existing thread will fork from that thread's history rather than the main conversation. This enables hierarchical task decomposition.

## Important Considerations

- **No audio output** вҖ” Threads only produce text responses via data messages. They cannot speak to the user.
- **Shared tools** вҖ” Threads use the same tool definitions as the parent, subject to any `toolFilter` applied at spawn time.
- **Stage changes** вҖ” All active threads are terminated when a [call stage](/agents/call-stages) change occurs.
- **Call end** вҖ” All threads are terminated when the call ends.
- **Thread IDs** вҖ” If you don't provide a `newThreadId`, one is auto-generated. Providing your own makes it easier to correlate responses.
